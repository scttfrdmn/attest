# ITAR on AWS

## The short version

**ITAR on AWS = AWS GovCloud.** Full stop.

AWS GovCloud (us-gov-west-1, us-gov-east-1) is the only AWS partition authorized
for ITAR-controlled technical data under 22 CFR §120.54. It is physically isolated
from AWS commercial regions and operated exclusively by US persons. The primary
technical control is a single SCP that denies all API calls outside GovCloud.

---

## What GovCloud provides

AWS GovCloud satisfies several ITAR requirements at the infrastructure level (AWS
responsibility under shared responsibility):

- **US-persons-only operations**: All AWS staff with logical or physical access to
  GovCloud are US citizens or permanent residents — screened and cleared by AWS
- **Physical isolation**: GovCloud data centers are separate from commercial AWS
  infrastructure
- **No foreign national access**: AWS's personnel controls prevent foreign national
  access to GovCloud, satisfying the export control requirement for "deemed exports"
  at the infrastructure level

AWS Artifact provides evidence of these controls. The customer responsibility is to
**stay in GovCloud** and **control who gets credentials**.

---

## The primary SCP

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "scp-itar-govcloud-only",
    "Effect": "Deny",
    "Action": ["*"],
    "Resource": "*",
    "Condition": {
      "StringNotEquals": {
        "aws:RequestedRegion": [
          "us-gov-west-1",
          "us-gov-east-1"
        ]
      }
    }
  }]
}
```

This SCP prevents any API call that would create or access resources in commercial
AWS regions — ensuring controlled technical data stays in GovCloud.

**Note**: This SCP conflicts with NIST 800-171's region restriction, which allows
`[us-east-1, us-west-2, us-gov-west-1]`. When ITAR and CMMC are both active on the
same org, **ITAR's restriction wins** — GovCloud only. Attest's conflict detection
handles this automatically, emitting the GovCloud-only SCP and suppressing the
broader NIST region SCP.

## The US person requirement

Beyond keeping data in GovCloud, ITAR requires that access to controlled technical
data be restricted to **US persons** (22 CFR §120.62):

- US citizens
- US lawful permanent residents (green card holders)
- Protected individuals under 8 U.S.C. §1324b(a)(3)

This is the part most institutions handle on paper. Attest enforces it at runtime
via a Cedar policy evaluating `principal.us_person`.

### How the attribute is sourced

The `us_person` determination lives in your HR/directory system — every institution
doing ITAR work already maintains this for their existing compliance programs. The
chain is:

```
HR System
  └── Maintains US person determination per employee/student
         │
         ▼
Institutional IdP (Shibboleth, Okta, Azure AD)
  └── Exposes us_person: true as a SAML/OIDC attribute
      (attribute release policy update — IT configuration change only)
         │
         ▼
Ark (AWS Research Kit) — optional but recommended
  └── Reads from IdP during researcher onboarding
      Writes: attest:us-person=true to IAM role tag
         │
         ▼
Attest principal resolver (SAMLSource)
  └── Reads attest:us-person from IAM role tags
      Populates: principal.us_person = true in Cedar evaluation
         │
         ▼
Cedar PDP (cedar-itar-us-person)
  └── Evaluates: principal.us_person == true
      ALLOW or DENY access to ITAR-controlled data
```

### Critical design principle: boolean only

The `us_person` attribute in attest is **always a boolean** — true or false.

Attest never stores, processes, or reasons about:
- Citizenship status
- Visa type or category
- Immigration status
- Country of birth or origin

The HR system makes the legal determination. Attest evaluates the derived boolean.
This is essential for FERPA (student data), employment law, and just basic
engineering hygiene — compliance systems should not store sensitive PII beyond
what they need.

## The Cedar policy

```cedar
// ITAR US person access control
// Auto-generated by attest — policy cedar-itar-us-person

// Principal must be a US person to access ITAR-controlled technical data.
// principal.us_person is sourced from institutional HR via the principal resolver.
forbid (
  principal,
  action,
  resource
)
unless {
  principal.us_person == true &&
  resource.itar_controlled == true
};
```

For mixed environments (some resources ITAR-controlled, some not), the
`resource.itar_controlled` attribute on the resource entity handles scoping — the
policy only applies to resources tagged as ITAR-controlled.

## ITAR + CMMC: the combined posture

For labs with DoD contracts involving both CUI and ITAR (common in defense-related
materials science, aerospace, and certain physics research):

| Control | CMMC/NIST 800-171 | ITAR | Combined |
|---|---|---|---|
| Region restriction | `[us-east-1, us-west-2, us-gov-west-1]` | `[us-gov-west-1, us-gov-east-1]` | **GovCloud only** |
| Encryption at rest | scp-require-kms-encryption | scp-require-kms-encryption | **Shared SCP** |
| MFA | scp-require-mfa | scp-require-mfa | **Shared SCP** |
| Audit logging | scp-protect-cloudtrail | scp-protect-cloudtrail | **Shared SCP** |
| Training | principal.cui_training_current | (training doesn't confer ITAR clearance) | **Separate** |
| Access control | principal.cui_training_current | principal.us_person | **Both required** |

When both frameworks are compiled together, attest produces:
- One `scp-itar-govcloud-only` (overrides the NIST region SCP)
- Shared SCPs for encryption, MFA, audit (one each satisfying both frameworks)
- Separate Cedar policies for CUI data movement (training-gated) and ITAR access (identity-gated)

## Practical note: ITAR environments are often separate orgs

In most institutions, ITAR research runs in a **separate AWS Organization in GovCloud**,
not co-mingled with HIPAA or general CUI research in commercial accounts. This is the
recommended architecture:

```
Commercial Organization (o-abc123)
├── HIPAA enclave OU (us-east-1, us-west-2)
│   └── genomics-hipaa-lab
└── CUI enclave OU (us-east-1, us-west-2)
    └── materials-cui-lab

GovCloud Organization (o-def456)   ← separate org, separate billing
└── ITAR enclave OU (us-gov-west-1, us-gov-east-1)
    └── defense-research-lab
```

`attest init` handles both: run it once per organization. The `sre.yaml` files are
separate. Attest's multi-SRE support (planned for v1.0.0) will provide a consolidated
view across both orgs.

## The honest state of ITAR compliance today

Most institutions doing ITAR research on AWS are handling this at the attestation
level: the PI signs a form, the export control office signs off, and the assessor
accepts it on paper. The technical controls — staying in GovCloud, enforcing US
person access at the API level — are rarely implemented programmatically.

Attest implements what the paper process describes. If your institution already has:
- GovCloud accounts (or plans to move there)
- `us_person` in your institutional IdP

...then attest wires those together automatically. The gap between "we attest that
we comply" and "the system enforces compliance at runtime" closes without process change.
